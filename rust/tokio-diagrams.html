// Async I/O in Rust with Tokio #[tokio::main(flavor = "current_thread")] // ONE thread! async fn main() { let (d1, d2, d3) = tokio::join!( fetch_url("url1"), // Starts immediately fetch_url("url2"), // Starts immediately fetch_url("url3"), // Starts immediately ); } // Total: ~1.5 seconds (all concurrent!)
How does one thread do this?
While Task A waits for network response, the thread immediately switches to Task B. When B waits, switch to C. When A's data arrives, switch back to A. No time wasted! üéØ

// CPU-bound: Use spawn_blocking #[tokio::main] async fn main() { let handles: Vec<_> = (0..3).map(|i| { tokio::task::spawn_blocking(move || { expensive_computation(i) // Pure CPU work }) }).collect(); for h in handles { h.await.unwrap(); } } // Uses thread pool for CPU work

üéØ6. When to Use What?

Single-Thread Async

  • Pure I/O workloads
  • Simpler debugging
  • No race conditions
  • Lower memory usage
  • Testing environments

Multi-Thread Async

  • Production systems
  • Mixed I/O + CPU work
  • Maximum performance
  • High throughput needed
  • poof-style trading!

spawn_blocking

  • CPU-intensive tasks
  • Blocking system calls
  • File I/O (sometimes)
  • Legacy blocking code
  • Database queries
I/O Work: Actively processing
Blocked: Wasted time waiting
CPU Work: Pure computation
Async: Efficient task switching
The Golden Rules üéØ

1. I/O-Bound? Use async/await. Even single-thread async is 4x+ faster!
2. CPU-Bound? Use spawn_blocking or multi-threading. Async won't help.
3. Mixed Workload? Multi-threaded Tokio runtime (default).
4. Need to handle 10,000+ connections? Async is your only option!
